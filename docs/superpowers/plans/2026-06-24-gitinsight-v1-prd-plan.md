# GitInsight V1 PRD 阶段计划（中文）

## 1. 计划目标
在不扩需求的前提下，完成 GitInsight V1 的 PRD 与设计定稿，为后续实施计划与开发提供单一事实来源。

## 2. 输入约束（已确认）
1. 第一优先角色：团队管理者。
2. 看板形态：项目健康总览驾驶舱（9 卡）。
3. 仓库范围：GitHub 公开 + 私有仓库。
4. 时间窗口：支持切换，默认 90 天。
5. 分析模式：快速层先出 + 深度层补齐。
6. 鉴权方式：PAT。
7. 安全策略：PAT 仅会话使用，不持久化。
8. 触发方式：手动 + 定时。
9. 部署策略：先单租户，架构预留多租户扩展。
10. 核心业务目标：覆盖三类价值，第一优先识别效率瓶颈。

## 3. 输出物
1. `docs/superpowers/specs/2026-06-24-gitinsight-v1-design.md`
2. `docs/superpowers/plans/2026-06-24-gitinsight-v1-prd-plan.md`
3. `README.md`（仅同步当前真实能力与下一阶段范围）

## 4. 执行步骤
1. 对齐上下文与边界，冻结 V1 范围。
2. 产出 PRD：目标、场景、指标体系、架构、状态机、异常策略。
3. 进行文档自检：占位词、矛盾项、歧义项、范围漂移。
4. 请求用户评审并收敛修改。
5. 评审通过后切换实施计划编写阶段。

## 5. 风险与缓解
1. 指标口径理解偏差：在 PRD 中逐条给出口径定义与示例字段。
2. 性能与体验冲突：采用分层分析和快照读模型。
3. 私有仓库权限失败：状态机区分可重试与不可恢复错误。

## 6. 验收标准
1. PRD 可直接作为实施输入，无关键 TBD/TODO。
2. 指标、接口、状态机、安全策略均有明确描述。
3. 与用户确认结论一致，不新增未确认需求。
