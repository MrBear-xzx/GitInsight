# GitInsight 质量规范（V1）

## 1. 文档目标
本规范定义 GitInsight V1 的质量基线、测试门禁、发布门禁与运行期质量指标，用于指导开发、测试、评审、上线与回归。

## 2. 质量原则
1. 先正确，再快速：核心指标口径正确性优先于展示速度优化。
2. 先可观测，再扩能力：所有关键链路必须可追踪可诊断。
3. 先防回归，再加功能：修复缺陷必须附带回归测试。

## 3. 质量分层定义
## 3.1 代码质量
1. 必须通过格式化与静态检查（按技术栈配置）。
2. 禁止引入未使用依赖与明显死代码。
3. 公共函数与关键模块需具备最小注释说明输入输出约束。

## 3.2 功能质量
1. 任务状态机流转正确且幂等。
2. 指标计算结果可复现，重复运行误差在可接受范围内。
3. 同一仓库同一时间窗口下，手动与定时触发结果口径一致。

## 3.3 数据质量
1. 指标输出字段完整，不允许关键字段为空。
2. 快照版本可追溯（仓库、时间窗、生成时刻）。
3. 缺失数据必须显式标注，不得静默置零。

## 3.4 运行质量
1. 快速层 P90 时延 <= 60 秒（中等规模仓库）。
2. 深度层 P90 时延 <= 10 分钟（中等规模仓库）。
3. 任务失败率（24h）<= 5%，核心失败码可聚合查询。

## 4. 测试策略与覆盖要求
## 4.1 TDD 执行要求
1. 新功能：先新增失败测试，再实现通过。
2. Bug 修复：先补失败回归测试，再修复。
3. 禁止以跳过测试代替问题修复。

## 4.2 覆盖范围要求
1. 单元测试：核心计算逻辑、风险分级逻辑、阈值边界。
2. 集成测试：任务创建、状态流转、快照写入与读取。
3. 端到端测试：仓库输入到看板可读的主路径。
4. 非功能测试：性能压测与限流/超时场景演练。

## 4.3 最低覆盖门槛（V1）
1. 核心指标计算模块行覆盖率 >= 85%。
2. 调度与状态机模块行覆盖率 >= 80%。
3. API 处理层行覆盖率 >= 75%。

## 5. CI 质量门禁
1. 单元测试必须全绿。
2. 集成测试必须全绿。
3. 关键 E2E 冒烟用例必须通过。
4. 静态检查必须通过。
5. 文档检查必须通过（接口字段与错误码示例齐全）。

任一门禁失败，禁止合并 PR。

## 6. 发布门禁
1. 最近一次主分支 CI 全绿。
2. 关键告警清零（PAT 泄露、任务雪崩、数据损坏）。
3. 版本发布说明包含变更范围、回滚方案、已知风险。

## 7. 缺陷分级与响应 SLA
1. P0：系统不可用或安全事故，30 分钟内响应，4 小时内给处置方案。
2. P1：核心流程不可用，2 小时内响应，24 小时内修复或降级。
3. P2：重要功能受损，8 小时内响应，3 个工作日内修复。
4. P3：一般问题，按迭代排期处理。

## 8. 质量度量看板（建议）
1. 任务成功率（日/周）。
2. 快速层与深度层 P90 时延。
3. 失败码分布与趋势。
4. 发布后 7 天回归缺陷数。
5. 指标口径变更次数与影响范围。

## 9. 安全质量规范
1. PAT 禁止持久化存储。
2. 日志中 PAT 必须掩码，默认只显示前后 2 位。
3. 错误响应不返回内部堆栈与敏感环境变量。
4. 安全问题修复必须附带回归验证脚本或测试。

## 10. 文档质量规范
1. README 仅描述当前真实能力，不写未实现能力。
2. SPEC 与 PRD 不一致时，以已评审通过的 SPEC 为准并回写 PRD。
3. 接口变更必须同步 `/docs` 示例。
4. 每次任务必须维护 `docs/superpowers/plans/` 下的中文计划文档。
