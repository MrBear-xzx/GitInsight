# GitInsight V1 产品需求与设计文档（PRD）

## 1. 文档信息
- 文档名称：GitInsight V1 PRD
- 文档日期：2026-06-24
- 文档状态：待评审
- 第一优先角色：团队管理者
- 范围版本：V1（单租户优先，架构预留多租户扩展）

## 2. 产品概述
GitInsight 是一个 Web 应用。用户输入 GitHub 仓库地址后，系统自动拉取并分析提交历史，生成可视化统计看板，帮助团队管理者识别效率瓶颈、交付风险和协作改进点。

V1 的核心定位是“项目健康总览驾驶舱”，默认展示 9 个核心指标卡，并支持快速首屏与深度补全的分层分析体验。

## 3. 目标与非目标
### 3.1 目标
1. 支持 GitHub 公开仓库与私有仓库分析。
2. 管理者可在同一看板观察团队效能、代码演进与项目健康度。
3. 首屏尽快可用：先返回快速指标，再异步补齐深度指标。
4. 支持手动触发和定时触发分析任务。
5. 默认时间窗口 90 天，并支持 30/90/180 天切换。

### 3.2 非目标（V1 不做）
1. 不支持 GitLab、Gitee 等多平台接入。
2. 不支持 PAT 持久化存储。
3. 不支持多租户完整权限体系。
4. 不提供复杂 BI 自定义拖拽报表。

## 4. 目标用户与核心场景
### 4.1 目标用户
- 团队管理者（第一优先）
- 技术负责人（次优先）
- 开源维护者（次优先）

### 4.2 核心场景
1. 每周/每月复盘时查看项目健康分、效率趋势和风险热点。
2. 发现评审链路变慢时快速定位瓶颈环节（响应慢或完成慢）。
3. 发现关键模块过度依赖少数成员时，推动任务再分配与知识扩散。

## 5. 功能范围（V1）
### 5.1 仓库接入
1. 用户输入 GitHub 仓库地址。
2. 私有仓库通过用户输入 PAT 访问。
3. PAT 仅会话内使用，不落库、不持久化。

### 5.2 分析触发
1. 手动触发：用户点击立即分析。
2. 定时触发：按配置周期自动创建任务。
3. 两种触发共享同一指标计算口径与结果存储结构。

### 5.3 看板展示
1. 首页为“项目健康总览驾驶舱”。
2. 默认 9 个核心指标卡，展示当前值、趋势、风险状态。
3. 指标支持 30/90/180 天时间窗口切换，默认 90 天。

### 5.4 分层分析
1. 快速层：优先输出轻量指标，保证首屏可读。
2. 深度层：后台补齐热点、协作网络与风险类指标。
3. 前端按任务阶段自动刷新展示。

## 6. 指标体系（9 卡）
### 6.1 卡片清单与口径
1. 交付吞吐（Delivery Throughput）
   - 口径：窗口内按日/周统计有效提交数趋势。
2. 变更前置时长（Lead Time Proxy）
   - 口径：代码首次提交到主干合入的中位时长（基于提交与合入时间代理）。
3. 评审响应时长（Review Response Time）
   - 口径：发起变更到首次评审反馈的中位时长。
4. 评审完成时长（Review Completion Time）
   - 口径：发起变更到完成评审并合入的中位时长。
5. 活跃贡献者数（Active Contributors）
   - 口径：窗口内提交次数达到阈值的贡献者人数。
6. Bus Factor 风险指数
   - 口径：关键模块贡献集中度（Top N 贡献占比）映射风险等级。
7. 热点文件波动指数（Hotspot Volatility）
   - 口径：高频变更文件的变更次数、作者离散度和短期抖动综合评分。
8. 返工率代理指标（Rework Ratio）
   - 口径：短时间内同文件重复修改或回滚相关提交占比。
9. 项目健康总分（Health Score）
   - 口径：前 8 项指标归一化加权总分，默认权重偏向效率链路。

### 6.2 风险分级
- 绿：低风险，趋势稳定或改善。
- 黄：中风险，指标波动超预警阈值。
- 红：高风险，连续恶化或超过上限阈值。

## 7. 系统架构设计
### 7.1 分层架构
1. Web 层
   - 功能：仓库录入、任务触发、任务状态查询、看板展示。
2. 调度层
   - 功能：任务入队、阶段编排、重试策略、定时任务触发。
3. 分析层
   - 快速分析器：轻量指标计算。
   - 深度分析器：热点、协作、风险指标计算。
4. 存储层
   - 任务表：记录状态机、触发来源、错误码。
   - 快照表：存放按仓库与时间窗口版本化的指标结果。

### 7.2 部署策略
1. V1 以单租户部署为主。
2. 数据模型预留 `organization_id`、`project_id` 维度字段，支持后续多租户演进。

## 8. 数据流与状态机
### 8.1 端到端数据流
1. 用户提交仓库地址与可选 PAT。
2. 系统创建分析任务并入队。
3. 执行快速层并写入 `snapshot_v1_quick`。
4. 前端展示首屏已就绪指标。
5. 深度层后台执行并写入 `snapshot_v1_full`。
6. 前端收到完成信号后刷新完整看板。

### 8.2 任务状态机
- `PENDING`
- `RUNNING_QUICK`
- `QUICK_DONE`
- `RUNNING_DEEP`
- `SUCCEEDED`
- `FAILED_RETRYABLE`
- `FAILED_FINAL`

### 8.3 异常处理
1. 克隆失败：进入可重试失败并指数退避。
2. API 限流：降级为本地日志可得指标，并标注部分缺失。
3. PAT 无效：进入不可恢复失败并提示重新输入。
4. 深度层超时：保留快速层结果并提示延迟补全。

## 9. 接口与文档要求
1. 对外接口变更必须在 `/docs` 可见。
2. 每个指标字段需包含定义、单位、取值范围与示例。
3. 任务接口需返回状态、耗时、错误码、缺失指标列表。

## 10. 安全与合规
1. PAT 只在会话上下文与任务执行时使用。
2. 日志中禁止打印 PAT 明文。
3. 数据库存储脱敏任务信息，不存 PAT 原文。
4. 失败回包不泄露内部栈信息或敏感环境变量。

## 11. 可观测性
1. 记录任务阶段耗时：克隆、快速分析、深度分析、写入。
2. 记录失败类型：网络、限流、鉴权、解析异常。
3. 提供任务运行审计：触发来源、执行时间、结果版本。

## 12. 测试与质量门禁
### 12.1 TDD 策略
1. 先写失败测试，再实现，再回归。
2. 修复类变更必须补回归测试。

### 12.2 测试分层
1. 单元测试：9 项指标计算口径与边界值。
2. 集成测试：任务状态流转与快照写入。
3. 端到端测试：仓库接入到看板渲染闭环。

### 12.3 门禁要求
1. 每次改动至少执行相关定向测试。
2. 提交前执行全量测试（可行时）。
3. CI 未通过不得合并。

## 13. 里程碑与验收标准
### 13.1 里程碑
1. M1：仓库接入与任务框架可运行（手动触发）。
2. M2：快速层指标与首屏驾驶舱可用。
3. M3：深度层指标、定时任务与风险提示可用。
4. M4：测试补齐、文档完善、CI 稳定通过。

### 13.2 验收标准
1. 可分析 GitHub 公开与私有仓库。
2. 首屏可在快速层完成后展示有效指标。
3. 深度层完成后可补齐 9 项指标并可切换时间窗口。
4. PAT 不持久化且日志不泄露敏感信息。
5. `/docs` 可见接口与字段说明，README 与实现能力一致。
