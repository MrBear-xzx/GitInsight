# GitInsight 协作代理规范

## 1. 语言与沟通
1. 默认使用中文沟通。
2. 回答遵循“先结论，后细节”，保持简洁可执行。
3. 在提交前、合并前、安装依赖前，必须主动同步并请求确认。

## 2. 分支与提交流程
1. 每个功能或修复使用独立分支：`feature/*`、`fix/*`、`chore/*`。
2. 标准流程：`main` 拉分支 -> 开发 -> 测试 -> 文档更新 -> 提交确认 -> push -> PR -> CI 通过 -> 合并 -> 回到 `main`。
3. 默认走 PR，不直接推送 `main`，除非用户明确授权。
4. 合并默认 `Squash and merge`，并删除远程分支。

## 3. 提交规则
1. 提交前必须给出改动摘要（改了什么、为什么、如何验证）。
2. 提交前必须给出建议中文 commit message，并等待用户确认。
3. commit 标题格式：`<type>(<scope>): <中文说明>`。

## 4. 质量与测试
1. 遵循 TDD：先写失败测试，再实现，再回归。
2. 每次改动至少跑定向测试；提交前可行时跑全量测试。
3. 修复类改动必须包含回归测试。
4. CI 未通过不得合并。

## 5. 文档要求
1. 每个可交付功能同步更新 `README.md`，且只描述真实能力。
2. 每个任务新增或更新 `docs/superpowers/plans/` 下中文计划文档。
3. 接口变更需保证 `/docs` 可见，并补字段说明与示例。

## 6. 安全与边界
1. 不擅自扩需求，不做无关重构。
2. 禁止破坏性命令（如 `git reset --hard`）除非用户明确授权。
3. 发现工作区异常改动需先暂停并询问用户。

## 7. 环境与工具偏好
1. 默认按 PowerShell 习惯执行命令。
2. 文本检索优先 `rg`。
3. `gh` 不在 PATH 时使用：`D:\Misc\gh_2.94.0_windows_amd64\bin\gh.exe`。
4. 需要安装依赖前必须先征得用户确认。

## 8. 任务收尾输出
每次任务收尾固定输出：
1. 改动清单
2. 测试结果（定向/全量）
3. 风险与影响范围
4. 下一步建议（可选）
